# Поток — финтех-сервис управления денежным потоком

## Идея в одном абзаце

**Поток** — финтех-сервис для фрилансеров и малого бизнеса, который показывает не «сколько денег сейчас», а **сколько будет через месяц**. Сервис строит прогноз баланса вперёд на основе повторяющихся доходов и расходов и выставленных счетов, заранее подсвечивает кассовый разрыв, автоматически откладывает резерв на налоги с каждого поступления и ведёт инвойсы. Основной оффер — «видеть деньги наперёд: ты знаешь, хватит ли тебе через три недели, ещё до того, как это станет проблемой».

Целевая аудитория: самозанятые, фрилансеры и микробизнес (1–10 человек), у которых деньги приходят неравномерно — проектами, инвойсами, с задержками, — и которым обычные банковские приложения показывают только прошлое.

## Почему такой стек

Продукт по природе — контентный маркетинг плюс насыщенный дашборд с данными и графиками. Здесь важны и SEO, и скорость первого экрана, и богатый интерактив. Один инструмент закрывает всё это без зоопарка:

- **Next.js (App Router)** — маркетинг рендерится статически (SEO, мгновенный первый экран), дашборд — на серверных компонентах с «островами» интерактива. Одна кодовая база, один деплой.
- **React Server Components** — данные тянутся на сервере, на клиент уезжает минимум JS. Для финтеха с тяжёлыми графиками это прямая экономия на бандле.
- **TypeScript (strict)** — деньги не прощают ошибок в типах; строгий режим ловит их до прода.
- **Tailwind + дизайн-токены** — единая система цветов, отступов и типографики, никакого разъезжающегося CSS.
- **TanStack Query** — серверное состояние, кэш и фоновая ревалидация балансов.

Итого: один фреймворк на маркетинг и продукт, лёгкий клиент, быстрые страницы и предсказуемый код.

## Фичи (MVP → v1)

**MVP:**
1. Лендинг с оффером, тарифами и соцдоказательством (SSG, под SEO).
2. Регистрация и рабочее пространство пользователя.
3. Дашборд: текущий баланс, **прогноз баланса на 30 дней**, статус кассового разрыва.
4. Ближайшие поступления и списания списком.
5. Инвойсы: список, статусы, создание счёта (форма с валидацией).
6. Категории расходов с разбивкой.

**v1:**
7. **Резерв на налоги** — виджет, который откладывает заданный процент с каждого поступления.
8. **Аналитика** — динамика дохода по месяцам, топ категорий расходов, интерактивные графики.
9. Повторяющиеся правила (аренда, подписки, регулярные клиенты) как основа прогноза.
10. Мультивалютность и локализация (₸ / ₽ / $), корректное форматирование сумм и дат.

## Полная архитектура

### Компоненты

```
┌─────────────────────────────────────────────────────────────┐
│                        Пользователь                          │
│        (десктоп и мобайл — большая часть с телефона)         │
└─────────────────────────────┬───────────────────────────────┘
                              │ HTTPS
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       Next.js на Vercel                      │
│  ┌───────────────┐  ┌────────────────┐  ┌────────────────┐   │
│  │  Маркетинг    │  │  Дашборд       │  │  Route          │  │
│  │  SSG + ISR    │  │  RSC + острова │  │  Handlers /     │  │
│  │  (лендинг)    │  │  интерактива   │  │  BFF-прокси     │  │
│  └───────────────┘  └────────────────┘  └────────────────┘   │
│         next/image · next/font · streaming · Suspense        │
└─────────────────────────────┬───────────────────────────────┘
                              │ типизированный API-клиент
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                 Backend API (вне зоны фронта)                │
│     аккаунты · транзакции · инвойсы · прогнозный движок      │
└─────────────────────────────────────────────────────────────┘
```

Фронт — самостоятельный слой: маркетинг и дашборд живут в Next.js на Vercel, а данные приходят из отдельного API через типизированный клиент. Чувствительные вызовы проксируются через Route Handlers, чтобы токены и ключи не утекали в браузер.

### Модель данных (контракт, который потребляет фронт)

Фронт не владеет схемой БД — он работает с типизированными ответами API. Ключевые сущности:

```
Account       id, balance, currency
Transaction   id, amount, direction (in|out), category, date, recurring
RecurringRule id, amount, direction, category, cadence (weekly|monthly), nextDate
Invoice       id, clientName, amount, currency, status (draft|sent|paid|overdue), dueDate
Forecast      points: [{ date, projectedBalance }], gapRisk (none|low|high)
TaxReserve    percent, accrued, currency
```

`Forecast` — производная сущность: движок на бэке считает проекцию баланса из транзакций, повторяющихся правил и неоплаченных инвойсов, а фронт получает готовый ряд точек и просто его рисует. Все суммы приходят в минимальных единицах (целые), форматирование в валюту — задача фронта через `Intl.NumberFormat`.

### Ключевые потоки

**1. Прогноз баланса**

```
серверный компонент дашборда
   ▼
запрашивает Forecast у API (на сервере, до отдачи HTML)
   ▼
рисует метрики (баланс, прогноз через 30 дней, риск разрыва) сразу
   ▼
график баланса — клиентский «остров», подгружается через dynamic import
   ▼
Suspense показывает скелетон, пока грузится тяжёлый чарт
```

Пользователь видит цифры мгновенно, а тяжёлая визуализация не тормозит первый экран и не тянет свой вес в общий бандл.

**2. Резерв на налоги (v1)**

Пользователь задаёт процент → с каждого входящего поступления виджет показывает, сколько отложено и сколько останется «чистыми». Значения пересчитываются на клиенте оптимистично и подтверждаются ответом API — никакого ожидания и мигания цифр.

**3. Создание инвойса**

Форма на React Hook Form + Zod: валидация на лету, понятные ошибки, блокировка повторной отправки. После успеха список инвойсов обновляется оптимистично, а прогноз баланса ревалидируется — новый счёт сразу влияет на проекцию.

### Структура проекта (App Router)

```
src/
  app/
    (marketing)/            лендинг, тарифы — SSG + ISR
    (dashboard)/
      overview/             прогноз, метрики, ближайшие движения
      invoices/             список и создание счетов
      analytics/            графики дохода и расходов
      layout.tsx
    api/                    Route Handlers: прокси к backend, вебхуки
  components/
    ui/                     примитивы: Button, Input, Card, Chart
    features/               доменные блоки: ForecastChart, InvoiceForm, TaxReserve
  lib/
    api/                    типизированный клиент + запросы TanStack Query
    hooks/
    format/                 валюты, даты, числа, i18n
  styles/tokens.css         дизайн-токены
```

Страницы разложены по группам маршрутов: маркетинг и продукт не мешают друг другу и рендерятся по-разному.

### Фронтенд-подход

Это ядро проекта — вакансия «в упор во фронт», и вот на чём стоит интерфейс:

- **Server Components по умолчанию, клиент — точечно.** Данные тянутся на сервере, интерактивными («островами») делаем только то, что действительно требует состояния: графики, формы, переключатели. Клиентский бандл остаётся лёгким.
- **Стриминг и скелетоны.** Через Suspense быстрые части экрана появляются сразу, тяжёлые (графики, аналитика) догружаются, показывая скелетон. Нет «белого экрана» и скачков раскладки.
- **Дизайн-система на токенах.** Цвета, отступы, радиусы и типографика заданы CSS-переменными; компоненты собираются из примитивов, ничего не хардкодится. Каждый компонент задокументирован в Storybook.
- **Mobile-first адаптив.** Вёрстка от мобильного вверх, флюидная типографика через `clamp()`, тач-таргеты от 44px, нижняя навигация на телефоне. Финтехом пользуются с телефона — это не «добавим потом».
- **Доступность.** Семантическая разметка, навигация с клавиатуры, управление фокусом, ARIA, контраст по WCAG AA.
- **Данные и деньги без сюрпризов.** Форматирование через `Intl.NumberFormat`, маскирование счетов, никаких прыгающих цифр, корректные знаки и валюты.

### Стратегия рендеринга

- **Лендинг и тарифы** — статическая генерация с инкрементальной ревалидацией: быстро и хорошо индексируется.
- **Дашборд** — серверные компоненты тянут данные, клиентские острова дают интерактив; тяжёлое грузится по требованию.
- **Страницы за авторизацией** — серверный рендер с проверкой сессии до отдачи HTML.
- **Живой баланс** — фоновая ревалидация через TanStack Query, чтобы цифры оставались свежими без ручного обновления.

### Производительность

Явные цели по Core Web Vitals: **LCP < 2.5 с, INP < 200 мс, CLS < 0.1**. Как достигаются:

- RSC — меньше JavaScript на клиенте.
- Код-сплиттинг по маршрутам и по компонентам: графики и формы через `dynamic()`.
- `next/image` и `next/font` — оптимизированные картинки и шрифты без layout shift.
- Префетч ссылок и кэширование на edge.
- Анализ бандла (`@next/bundle-analyzer`) и **Lighthouse CI** в пайплайне на каждый pull request — регрессия по скорости не проходит ревью незамеченной.

### Безопасность и приватность

- Токены сессии в HttpOnly-cookie, чувствительные вызовы — через серверные Route Handlers, ключи не попадают в браузер.
- Никаких персональных или финансовых данных в URL и query-параметрах.
- Маскирование номеров счетов в интерфейсе, аккуратное обращение с суммами.
- Строгий CSP и заголовки безопасности на уровне Next.js.

### Тарифы и юнит-экономика

- **Free** — один прогноз, базовый дашборд, до N транзакций. Крючок активации.
- **Pro** — полный прогноз, инвойсы, резерв на налоги, аналитика.
- **Business** — мультивалютность, командный доступ, приоритетная поддержка.

Фронт на Vercel дёшев в обслуживании: статика раздаётся с edge, дашборд — RSC без тяжёлого рантайма. Основные расходы на стороне backend-движка, фронт масштабируется почти бесплатно.

### Дорожная карта

- **MVP** — лендинг, дашборд с прогнозом, ближайшие движения, инвойсы, категории расходов.
- **v1** — резерв на налоги, аналитика, повторяющиеся правила, мультивалютность.
- **v2** — сценарии «что если» (добавить клиента / убрать расход и увидеть новую проекцию), напоминания по неоплаченным счетам, экспорт для бухгалтера.

### Метрики успеха

- **Активация** — пользователь увидел свой первый прогноз баланса (значит, данные подключились и ценность считана).
- **Удержание** — доля пользователей, активных на 4-й неделе.
- **Главная метрика ценности** — сколько кассовых разрывов сервис подсветил заранее; это ровно та проблема, ради которой продукт существует.

### Показатели фронта (то, чем меряется качество для этой вакансии)

- Lighthouse Performance 90+ на дашборде и лендинге.
- Core Web Vitals в зелёной зоне на мобильном.
- Ноль критичных нарушений доступности в axe.
- Покрытие ключевых компонентов тестами (Vitest + Testing Library) и пользовательских сценариев (Playwright).
